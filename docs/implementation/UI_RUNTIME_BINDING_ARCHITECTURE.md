# UI Runtime Binding Architecture

Status: Implementation bridge draft.
Issue: #21.

## Core Rule

The UI is continuity visualization. It must display implemented runtime state, not decorative abstractions.

## Binding Targets

- Continuity stages map to screen state.
- Continuity reports map to player-facing panels.
- Archive records map to replay, Verdict, Trial, Gold, Sophia, and audit history views.
- Replay reconstruction state maps to replay viewer validation indicators.
- Divergence and recovery state map to synchronization and recovery UI.
- Gold observation output maps to non-intervention pattern panels.
- Sophia interpretation output maps to guidance and survivability panels.

## Continuity Stage Screen Mapping

- Construct: factory, preparation, and pressure-environment creation screens.
- Synchronize: continuity search, matchmaking, and participant readiness screens.
- Audit: trace activation, lock, and authority confirmation screens.
- Execute: match and active runtime screens.
- Return: post-match return and processing screens.
- Replay: replay viewer and reconstruction screens.
- Verdict: consequence reconstruction and puzzle-analysis screens.
- Trial: gate, fragment, and pressure-environment training screens.
- Archive: library, memory, report, and artifact browser screens.

## Material Binding

UI material must represent runtime behavior.

- Regulate: audit, locks, rules, Gold, and validated reports.
- Operate: buttons, available actions, and execution controls.
- Secure: hidden, locked, restricted, or recovery-sensitive state.
- Transfer: movement, synchronization, routing, archive references, and ownership flow.

## Report Binding

ContinuityReport outputs should appear as structured UI panels. Reports may be generated by Gold, Sophia, audit systems, Trial systems, Verdict systems, or archive systems.

## Implementation Rule

Every major UI panel should identify the runtime source it visualizes. If a panel has no runtime source, it must remain concept-only until a source exists.
