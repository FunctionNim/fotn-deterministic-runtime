# Branch Protection Setup Guide — R10

This document records the recommended manual branch-protection policy for `FunctionNim/fotn-deterministic-runtime`.

The GitHub connector can inspect and edit repository files, but branch protection is a GitHub repository setting that should be confirmed in the GitHub UI by an administrator.

## Purpose

The deterministic runtime is protected by local verification, regression tests, golden snapshots, audit manifests, and GitHub Actions. Branch protection turns those protections into an enforced merge rule for `main`.

## Protected Branch

```text
main
```

## Recommended GitHub UI Path

In GitHub:

```text
Repository → Settings → Branches → Branch protection rules → Add branch protection rule
```

Use this branch name pattern:

```text
main
```

## Required Status Checks

Enable:

```text
Require status checks to pass before merging
```

Require the runtime verification workflows that currently protect the project:

```text
Verification Gate
```

```text
deterministic-runtime-ci
```

If GitHub shows more specific job names under either workflow, select the concrete jobs that run the following checks:

```bash
npm run build
npm test
dotnet build
npm run demo
```

## Up-to-Date Branch Requirement

Recommended:

```text
Require branches to be up to date before merging: enabled
```

Purpose: make sure the verification gate runs against the current `main` state before a pull request is merged.

## Pull Request Requirement

Recommended once active collaboration begins:

```text
Require a pull request before merging: enabled
```

Solo-builder mode may temporarily allow direct pushes, but the repository should eventually move to pull-request-based changes once the runtime stabilizes further.

## Direct Push Policy

Current solo-build recommendation:

```text
Direct pushes to main: allowed only for trusted maintainer work
```

Future collaboration recommendation:

```text
Direct pushes to main: disabled
All changes enter through pull requests
```

## Golden Snapshot Policy

Branch protection should preserve the snapshot rule:

```text
CI must never regenerate golden snapshots.
```

Golden snapshots may only be updated intentionally by a maintainer after reviewing the behavior change.

Intentional snapshot update command:

```bash
npm run generate-snapshot
```

Normal verification command chain:

```bash
npm run build
npm test
dotnet build
npm run demo
```

## Recommended Additional Settings

Recommended when available:

```text
Require conversation resolution before merging: enabled
```

```text
Require linear history: optional
```

```text
Include administrators: optional during solo-build, recommended for stricter release discipline
```

```text
Allow force pushes: disabled
```

```text
Allow deletions: disabled
```

## Manual Enforcement Checklist

Use this checklist after enabling or reviewing branch protection:

```text
[ ] Branch rule exists for main.
[ ] Required status checks are enabled.
[ ] Verification Gate is required.
[ ] deterministic-runtime-ci is required.
[ ] Branches must be up to date before merging.
[ ] Force pushes are disabled.
[ ] Branch deletion is disabled.
[ ] Pull requests are required when collaboration begins.
[ ] Golden snapshots are not regenerated by CI.
[ ] README links to this policy.
```
